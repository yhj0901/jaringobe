# API 백엔드 시작 — jaringobe

## 사용 예시
```
# 기본 (설계문서 자동 로드)
/API시작

# 특정 도메인만 구현
/API시작 fridge 도메인 재고 차감 API만 먼저 구현

# 기존 코드 버그 수정
/API시작 [버그] 식단 조회 API 페이지네이션 누락

# 기존 코드 고도화
/API시작 [고도화] 냉장고 재고 차감 쿼리 최적화

# 기존 코드 기능 변경
/API시작 [변경] 자동 주문 감산 로직에 유통기한 가중치 추가

# DB 변경 포함
/API시작 [변경] fridge_item 테이블 expires_at 인덱스 필요 / DB 변경 있음
```

> 이 커맨드는 오케스트레이터의 GATE 2 승인 후에만 실행한다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/backend.md` 파일을 읽고 API 백엔드 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```
입력이 없는 경우 자동으로 로드:
- `docs/설계/api-spec.md`
- `docs/설계/db-schema.md`
- `docs/설계/security-design.md`
- `docs/설계/store-adapter.md` (마트 연동 시)
- `docs/기획/` (유저 스토리 확인)

## 3. 브랜치 준비 (필수, 코드 수정 전)
- `git fetch` 후 **최신 `main` 기반으로 `feature/{기능}` 브랜치 생성**
- 이미 관련 브랜치가 있으면 `git merge main --ff-only` 로 최신화 후 이어서 작업

## 4. 기존 코드 분석
- `backend/app/api/v1/` 에서 유사 도메인 라우터 패턴 분석
- `backend/app/services/` 에서 서비스 레이어 패턴 분석
- `backend/app/schemas/` 에서 Pydantic 스키마 패턴 분석

## 5. 기능 단위 분리 계획
```
## API 백엔드 기능 단위 분리 계획
| 순서 | 브랜치명 | 구현 내용 | 도메인 | DB 변경 | 의존성 |
|------|---------|---------|-------|-------|-------|
| 1 | feature/{기능} | {내용} | {도메인} | 있음/없음 | 없음 |
```

## 6. 기능 단위별 구현
```
[각 기능 단위마다]
① 3단계에서 준비한 feature 브랜치에서 작업
② Pydantic 스키마 정의 (schemas/{도메인}.py)
③ SQLAlchemy 모델 (models/{도메인}.py — DB 변경 시 인프라 에이전트 연동, GATE 3)
④ 서비스 로직 (services/{도메인}/)
⑤ APIRouter (api/v1/{도메인}.py)
⑥ pytest 단위/통합 테스트
⑦ ruff + mypy PASS
⑧ 코드 리뷰 요청 → 머지
```

## 7. 완료 보고
```
## API 백엔드 구현 완료 보고
- 구현 완료 엔드포인트: {N}개
- 도메인: {목록}
- DB 변경: {있음/없음}
- 테스트 통과: {N}건 (커버리지 {N}%)
- 미결 사항: {있을 경우}
```

# API 수정 시작 (기존 코드)

$ARGUMENTS에 `[버그]`, `[고도화]`, `[변경]` 키워드가 포함된 경우
`agents/backend.md`의 **기존 코드 수정 에이전트** 섹션을 따른다.

```
① 코드 분석 → ② 영향도 분석 → ③ 수정 계획 보고
→ 오케스트레이터 확인 (리스크 High / DB 변경 시 필수)
→ ④ 구현 → ⑤ 회귀 테스트 → ⑥ 코드 리뷰 → 머지
```
