# UI 프론트엔드 시작 — jaringobe

## 사용 예시
```
# 기본 (설계문서 자동 로드)
/UI시작

# 특정 화면만 구현
/UI시작 가상 냉장고 재고 목록 화면만 먼저 구현

# 기존 코드 버그 수정
/UI시작 [버그] 예산 락 게이지가 통화 전환 시 깨짐

# 기존 코드 고도화
/UI시작 [고도화] 주간 식단 화면 렌더링 최적화

# 기존 코드 기능 변경
/UI시작 [변경] 식사 완료 체크에 취소(되돌리기) 추가
```

> 이 커맨드는 오케스트레이터의 GATE 2 승인 후에만 실행한다.

---

아래 순서로 진행한다.

## 1. 에이전트 하네스 로드
`agents/frontend.md` 파일을 읽고 UI 프론트엔드 에이전트 규칙을 완전히 숙지한다.

## 2. 입력 확인
```
$ARGUMENTS
```
입력이 없는 경우 자동으로 로드:
- `docs/설계/api-spec.md`
- `docs/설계/ui-design.md` (있을 경우)
- `docs/설계/architecture.md`
- `docs/기획/` (유저 스토리, UX 흐름 확인)

## 3. 브랜치 준비 (필수, 코드 수정 전)
- `git fetch` 후 **최신 `main` 기반으로 `feature/{기능}` 브랜치 생성**
- 이미 관련 브랜치가 있으면 `git merge main --ff-only` 로 최신화 후 이어서 작업

## 4. 기존 코드 분석
- `frontend/src/features/` 에서 유사 도메인 컴포넌트 패턴 분석
- `frontend/src/lib/api/` 에서 API 클라이언트 패턴 분석
- `frontend/src/messages/` 에서 i18n 키 네이밍 패턴 확인

## 5. 기능 단위 분리 계획
```
## UI 프론트엔드 기능 단위 분리 계획
| 순서 | 브랜치명 | 구현 내용 | 라우트/컴포넌트 | API 의존 | 의존성 |
|------|---------|---------|---------------|---------|-------|
| 1 | feature/{기능} | {내용} | {경로} | {엔드포인트} | 없음 |
```

## 6. 기능 단위별 구현
```
[각 기능 단위마다]
① 3단계에서 준비한 feature 브랜치에서 작업
② 컴포넌트 설계 (Props, 서버/클라이언트 구분)
③ API 클라이언트 + 타입 정의 (lib/api/, types/)
④ 페이지/컴포넌트 구현 (app/[locale]/, features/)
⑤ i18n 키 추가 (ko.json / en.json 동시)
⑥ vitest 단위 테스트
⑦ ESLint / Prettier / tsc PASS
⑧ 코드 리뷰 요청 → 머지
```

## 7. 완료 보고
```
## UI 프론트엔드 구현 완료 보고
- 구현 완료 화면/컴포넌트: {N}개
- i18n 키 추가: {N}개 (ko/en 동시)
- 테스트 통과: {N}건 (커버리지 {N}%)
- 미결 사항: {있을 경우}
```

# UI 수정 시작 (기존 코드)

$ARGUMENTS에 `[버그]`, `[고도화]`, `[변경]` 키워드가 포함된 경우
`agents/frontend.md`의 **기존 코드 수정 에이전트** 섹션을 따른다.

```
① 코드 분석 → ② 영향도 분석 → ③ 수정 계획 보고
→ 오케스트레이터 확인 (리스크 High 시 필수)
→ ④ 구현 (i18n 변경 시 ko/en 동시) → ⑤ 회귀 테스트 → ⑥ 코드 리뷰 → 머지
```
