# 버그 리포트 — 앱 웹뷰 래핑 + 푸시 알림 QA (2026-07-14)

**Critical 0 / High 3 / Medium 3 / Low 1 → GATE 4 불통과 (High 해소 후 회귀 재판정)**

## BUG-001 [High] [API]/[설계] — 좀비 processing 플랜이 식단 생성을 영구 차단
- 재현: `POST /mealplans` 202 접수 → BackgroundTasks(인프로세스) 실행 중 서버 재시작 → 플랜 `processing` 영구 잔류
- 기대: 중단된 생성이 결국 failed 로 수렴, 재시도 가능
- 실제: `_ensure_not_generating`(mealplan/service.py:238-246)이 시간 컷오프 없이 409 → 신규 생성·재생성 영구 409. 프론트는 409→진행 플랜 폴링 합류→3분 timeout→재시도해도 재합류 — UI 로 벗어날 수 없는 잠금
- 수정 방향: processing stale 판정(created_at + N분 초과 시 failed 취급) + 기동 시 정리. `_mark_generation_failed` 실패 경로도 동일 수렴 보장

## BUG-002 [High] [API] — failed 플랜 재생성 시 mealsPerDay 가 1 로 붕괴
- 재현: days=3·mealsPerDay=3 생성 실패(failed) → 홈 재시도 배너 → `POST /{id}/regenerate`
- 기대: 원 요청(하루 3끼) 유지
- 실제: mealplan/service.py:422 `max(1, round(len(plan.meals)/days))` — failed 플랜은 meals 0개 → 하루 1끼 플랜 생성 (루프 ① 조용한 왜곡). 원 요청 mealsPerDay 미저장이 근본 원인
- 수정 방향: 접수 시 요청 파라미터(days/mealsPerDay/allergies/preferences) 보존 후 재생성에 사용 (실패 후 재시도가 주 사용 경로)

## BUG-003 [High] [UI(mobile)] — 웹뷰 오리진 검증 프리픽스 매칭 우회 (CWE-345/601)
- 재현: `https://jaringobe.cloud.evil.com` 링크가 웹 내 1회 노출되면 성립
- 기대: WEB_URL 오리진 완전 일치만 내부 허용 (mobile-app.md 3·4장)
- 실제: mobile/src/webview.tsx:123,153 `url.startsWith(WEB_URL)` — 위 도메인이 내부 내비게이션 허용 + 해당 페이지 postMessage 가 브리지 메시지로 수리(LOGIN_PROVIDER 피싱 등)
- 수정 방향: URL 파싱 후 origin 완전 일치 (`url === WEB_URL || startsWith(WEB_URL + '/')` 이상 권장: new URL().origin 비교)

## BUG-004 [Medium] [API] — 앱 로그인 원타임 코드 이중 소진 race (CWE-362)
- 재현: 동일 코드 동시 2요청 (탐침으로 창 확대 시 실증). 정상 스케줄링에선 재현 희박, 이중 소진돼도 동일 유저 세션 2개 (교차 사용자 상승 없음)
- 실제: auth/service.py:196-211 `consume_app_login_code` 가 SELECT→확인→UPDATE 3단계, 행 락·조건부 UPDATE 없음
- 수정 방향: `UPDATE ... SET used_at=now() WHERE code_hash=:h AND used_at IS NULL RETURNING user_id` 원자 소진

## BUG-005 [Medium] [API] — 다중 플랜 공존 시 구플랜 끼니로 리마인더 오발송 (FR-006 위반)
- 재현: 신규 POST 로 새 플랜 생성(구플랜 ready 잔존, 기간 중첩) → 신플랜 점심 완료 → 리마인더가 구플랜 미완료 점심을 집어 발송 (탐침 실증)
- 실제: notification/scheduler.py:62-75 — completed_at 필터를 전 활성 플랜에 건 뒤 정렬 → 최신 플랜 완료 시 구플랜 행으로 폴백
- 수정 방향: 최신 활성 플랜을 먼저 선택 → 그 플랜 내에서만 끼니·완료 판정

## BUG-006 [Medium] [UI]/[설계] — 초기 브리지 메시지 유실 race
- 재현: 앱 onLoadEnd 시 BRIDGE_READY/PERMISSION_STATUS/PUSH_TOKEN 1회 발신(mobile/src/webview.tsx:167-173) vs 웹 리스너는 React effect 마운트 후 구독(BridgeProvider.tsx:14-31) — 하이드레이션 전 발신되면 유실
- 영향: 토큰 등록·soft ask·OS 배너·로그아웃 시 DELETE devices 가 조용히 누락
- 수정 방향: 웹→앱 `SYNC_REQUEST` 메시지 추가(구독 완료 후 요청) — 브리지 프로토콜 증보 필요 → 설계 변경 프로세스 경유 (mobile-app.md 3장 + frontend/mobile 양쪽)

## BUG-007 [Low] [UI] — 409 합류 시점에 이미 완료된 플랜을 실패로 표시
- 실제: generation.ts:103-110 — 409 후 latest 가 processing 일 때만 합류, ready/over_budget 면 failed 반환 → 성공을 실패 배너로 표시
- 수정 방향: ready/over_budget 면 completed 처리
