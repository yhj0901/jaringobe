# 예산 락 + AI 식단 자동 생성 기획서

## 개요 | 배경 및 목적
사용자가 **월 식비 예산을 통화와 함께 락(Budget Lock)** 하면, 가구 구성·알레르기·선호를 반영해 **예산을 초과하지 않는** 기간별 식단(끼니별 + 조리법)을 AI가 자동 편성하고, 남은 예산·절약액(예산 무드)을 보여준다.
사업계획서 3대 축 중 **①예산 내 식단 자동 생성**이며, 선순환 루프(입력 → 식단 → 냉장고 → 동적 감산)의 **출발점**이다.

## 범위: In Scope / Out of Scope
**In Scope (v1)**
- 월 식비 예산 설정/락 (통화 KRW/USD, 기간)
- 가구 구성·알레르기·선호 기반 **LLM 식단 자동 생성** (일자별 아침/점심/저녁 + 조리법)
- 예산 초과 방지: LLM 편성 → **백엔드 검산 → 초과 시 재시도 → 실패 시 투명 노출**
- 식단 조회 (일자별/끼니별)
- 식단 재생성 (전체 / 특정 끼니)
- 예산 무드 표시 (남은 예산·편성 비용·절약액) — 초안 수준
- 재료 가격은 **기준가 테이블(`ingredient_price_ref`)** 기반, `PriceProvider` 인터페이스로 격리

**Out of Scope (후속)**
- 실시간 마트 최저가 크롤링/연동 (store 도메인)
- 자동 주문·결제 (order 도메인)
- 가상 냉장고 재고 차감 반영 (fridge 도메인) — v1은 재고 0 가정
- 소셜 로그인 상세 (auth 도메인, 본 기능은 인증된 사용자 전제)
- 상세 영양 분석(칼로리/영양소 정밀), 프리미엄 목적별 식단(구독)
- LLM 생성 비동기화(큐/SSE) — v1은 동기

## 관련 도메인
- **주 도메인**: `budget`, `mealplan`
- **의존 도메인**: `household`(가구 구성 입력), `auth`(인증 전제)
- **후속 연계**: `store`(가격/주문), `fridge`(재고 반영), `order`(자동 주문)

## BM 연결점
- **구매 전환 중개 수수료(2~5%)**: 예산 내 식단 = 필요 식재료 목록 → 후속 자동 주문의 **진입점**
- **프리미엄 식단 월 구독**: 식단 방향(건강/다이어트/든든/아이 입맛) 옵션이 구독 상품(다이어트·벌크업 전용)으로 확장되는 기반

## 글로벌 고려 (KR/US 동작 차이, i18n, 통화)
- **통화**: KRW/USD 동시 지원. 금액은 `Decimal(numeric) + 통화코드` 저장 (float 금지)
- **i18n**: 모든 UI 문구 ko/en 키 관리. 레시피/재료명은 지역별 식문화 반영(기준가 테이블도 region별)
- **타임존**: 예산 기간(period) 경계·식단 일자는 UTC 저장, 사용자 로캘로 표시
- **지역별 식단**: region(KR/US)에 따라 LLM 프롬프트의 식문화 컨텍스트와 기준가 테이블이 달라짐

## 유저 스토리
- **US-001** 사용자는 월 식비 예산을 통화(KRW/USD)·기간과 함께 설정하고 락할 수 있다
- **US-002** 사용자는 가구 구성·알레르기·선호를 반영한 식단을 자동 생성받는다
- **US-003** 생성된 식단의 총 편성 비용은 설정한 예산을 초과하지 않는다 (초과 불가피 시 명시적 경고)
- **US-004** 사용자는 생성된 식단을 일자별/끼니별로 조회한다
- **US-005** 사용자는 식단 전체 또는 특정 끼니를 재생성할 수 있다
- **US-006** 사용자는 예산 대비 현재 편성 비용·남은 예산·절약액(예산 무드)을 확인한다

## 비기능 요구사항
- **성능**: 식단 생성 응답 P95 ≤ 25s (동기 + 타임아웃), 조회 P95 ≤ 300ms
- **비용**: LLM 결과 저장으로 재호출 최소화, 재생성만 재호출
- **가용성**: LLM 호출 타임아웃 + 재시도/폴백 (자동주문 루프 무중단 원칙 연계)
- **보안/개인정보**: 하단 참조. 접근성: 예산 무드 색상 대비(WCAG AA)

## UX 흐름 (화면 단위)
1. **예산 설정** — 금액/통화/기간 입력 → 락
2. **생성 로딩** — 스켈레톤 + 진행 표시 (수 초)
3. **식단 캘린더** — 주간 뷰 → 일자 선택 → 끼니 카드(아침/점심/저녁, 조리법 링크)
4. **예산 무드 바** — 편성 비용/남은 예산/절약액, 초과 시 경고 배너
5. **재생성** — 전체 또는 특정 끼니 카드 단위

## 보안 요구사항 (CWE 포함)
- **CWE-639 / CWE-284 (IDOR/인가 우회)**: 본인 `household`의 예산/식단만 접근. 모든 엔드포인트 인증 의존성 + 소유권 검증
- **CWE-20 (부적절한 입력 검증)**: 예산 금액 범위, 통화 코드 화이트리스트, 기간 유효성 — Pydantic 스키마
- **프롬프트 인젝션**: 사용자 선호/알레르기 자유 텍스트가 LLM 프롬프트에 유입 → 입력 정제·구조화, 시스템 지시 분리, **알레르기는 생성 후 코드로 재검증**(LLM 신뢰 금지)
- **CWE-770 (제한 없는 자원 할당)**: 식단 재생성 rate limit (LLM 비용/DoS 방어)
- **개인정보**: 알레르기 = **건강 민감정보**(GDPR 특수범주/국내 민감정보). 최소 수집, 보관기간 명시, 로그 미출력
- **금액 정합성**: `Decimal(numeric)` + 통화코드, float 금지

## 데이터 요구사항 (신규/변경 엔티티, 보관 기간)
| 엔티티 | 주요 필드 | 비고 |
|--------|-----------|------|
| `budget` | household_id(FK), amount(numeric), currency, period_start/end(timestamptz), locked(bool) | 신규 |
| `meal_plan` | household_id(FK), budget_id(FK), period, status, total_cost(numeric), currency | 신규 |
| `meal` | meal_plan_id(FK), plan_date, meal_type, recipe_name | 신규 |
| `meal_ingredient` | meal_id(FK), name, quantity, unit | 신규 |
| `ingredient_price_ref` | name, region, unit_price(numeric), currency, pack_qty, unit | 신규(기준가 seed) |
- 인덱스: `meal_plan(household_id, period)`, `meal(meal_plan_id, plan_date)`
- **DDL은 인프라 에이전트 전담(GATE 3)** — 마이그레이션 `backend/alembic/`
- 보관 기간: 식단/예산은 사용자 계정 존속 기간, 탈퇴 시 파기(개인정보 방침 연동)

## 기술 제약사항
- 백엔드 Python 3.12+ / FastAPI / SQLAlchemy 2.0(async, asyncpg) / Pydantic v2 / uv
- LLM 호출은 `backend/app/services/mealplan/`에 격리, 타임아웃 + 재시도/폴백
- 재료 가격은 `PriceProvider` 인터페이스 뒤로 격리(v1 기준가 테이블 → 후속 store 어댑터 교체)
- v1 식단 생성은 **동기** (비동기 큐/SSE는 후속)
- v1은 냉장고 재고 0 가정 (fridge 연계는 후속)
