# 게스트 홈 (진입 경험) 기획서

## 개요 | 배경 및 목적

자린고비의 **비로그인 첫 진입점**. 마케팅 랜딩 대신 **샘플 데이터가 채워진 실제 홈 화면(게스트 홈)** 을 먼저 보여줘 "이건 뭐하는 서비스지?"를 제품 그 자체로 답한다. 둘러보는 사용자를 끊지 않으면서, 체류 시점에 맞춰 **예산안 작성 → 예산 적용 홈 → 자동주문 제안 → 회원가입**으로 자연스럽게 잇는 점진적 전환 퍼널을 만든다.

- 배경: ① 첫 기획(로그인 우선)은 진입장벽 → ② 랜딩+체험 위젯으로 전환 → ③ 오케스트레이터 시나리오 구체화로 **게스트 홈 + 타이밍 프롬프트** 구조로 재전환 (2026-07-09, 본 문서가 `랜딩-예산체험.md` 를 대체)
- 핵심 시나리오:
  1. 첫 방문 → 샘플 데이터 게스트 홈을 자유롭게 둘러봄 ("체험 모드" 배지)
  2. 체류 약 10초 후 "예산안을 작성해 보시겠어요?" 바텀시트 — **아니오 → 그대로 계속 둘러보기** (더 확인하고 싶은 사용자 존중)
  3. 예 → 예산안 작성(인원·예산·식단 방향) → 홈 전체가 내 예산안 기반 샘플로 갱신 (여전히 게스트)
  4. 예산 적용 홈에서 자동주문 제안 → 마트 계정 연결 전 **회원가입(소셜 로그인)** → 게스트 예산안 계정 이전
- 화면 순서: **`/` 게스트 홈 → (예산안 작성 오버레이) → `/login` 소셜 로그인 → 예산안 이전(온보딩 스킵) → 로그인 홈**

## 범위

### In Scope
- 게스트 홈 (`/`): 실제 홈과 동일한 UI 셸 + 샘플 데이터 — 식단 카드(아침/점심/저녁), 예산 무드(남은 예산·절약·폐기 절감), 가상 냉장고 위젯, 자동주문 카드. "체험 모드" 배지 상시 노출
- 타이밍 프롬프트: 체류 10초(+스크롤 유휴) 시 "예산안을 작성해 보시겠어요?" 바텀시트 1회. 아니오 → 세션 내 재노출 금지 + 상단 상시 CTA 배너로 대체
- 예산안 작성 플로우 (게스트, 3스텝 이내): 가구 인원 → 월 예산(로캘별 프리셋+직접 입력) → 식단 방향(건강/다이어트/든든/아이 입맛)
- 예산안 적용: 홈 전체 위젯이 예산안 기반 **정적 샘플 매트릭스**로 일괄 갱신 (AI 호출 없음)
- 자동주문 전환 지점: 예산 적용 후 자동주문 카드 활성 + 1회 알림 → 진행 시 `/login` 이동
- 게스트 데이터 localStorage 보관(30일) — 재방문 시 예산안 복원
- 가입 후 게스트 예산안 계정 이전 + 온보딩 스킵 규칙 (접점 정의)
- 기타 쓰기 행동(식사 완료 체크, 전체 조리법 보기 등) 공통 가입 유도 모달
- SEO/OG 메타태그 (첫 마케팅 표면)

### Out of Scope
- 실제 AI 식단 생성 (게스트는 정적 샘플 매트릭스만, LLM 호출 없음 — 실제 생성은 가입 후 mealplan 기획)
- 로그인 페이지 자체 (`docs/기획/로그인-소셜인증.md`)
- 마트 계정 연결·실제 자동 주문 (store/order 별도 기획 — 본 기획은 가입 전환 지점까지)
- 로그인 후 홈의 실데이터 동작 (홈 UI 셸은 본 기획에서 구현, 실데이터 연결은 각 도메인 기획에서)
- 애널리틱스/실험(프롬프트 타이밍 최적화 포함) — 별도 기획

## 영향 영역
- [x] frontend (Next.js) — 게스트 홈 셸, 타이밍 프롬프트, 예산안 플로우, 샘플 매트릭스, localStorage
- [x] backend (FastAPI) — **가입 시 게스트 예산안 이전 수신 API 1건** (서버측 재검증 포함, 상세는 설계). 그 외 게스트 동작은 backend 무관
- [ ] DB (PostgreSQL 스키마 변경) — 없음 → **GATE 3 비대상** (예산안 저장처는 budget/household 도메인 기획·설계에서 확정 — 본 기획은 이전 계약만 정의)
- [ ] 마트 연동 (store 어댑터) — 없음 (전환 지점까지만)

## 관련 도메인
- budget (예산안 작성·예산 무드 — 중심), household (인원), mealplan (샘플 식단 표현), fridge (냉장고 위젯 프리뷰), order (자동주문 카드·전환 지점), auth (가입 게이트 접점)

## BM 연결점
- **전환 퍼널 전체를 제품 안에서 구성**: 둘러보기 → 예산안(투자 행동) → 자동주문 제안(수익원 직결) → 가입. 자동주문은 중개 수수료 BM 의 핵심 진입점
- 예산 무드의 절약·폐기 절감 지표를 가입 전에 체감 → 프리미엄 구독 가치 제안 선행 학습

## 글로벌 고려 (KR/US 동작 차이, i18n, 통화)
- 예산 프리셋 로캘별: ko → 30/50/70/100만원(KRW) / en → $300/$500/$700/$1,000(USD), 직접 입력 병행
- 샘플 매트릭스 로캘별 분리 (ko 한식 / en 미국 가정식), 자동주문 카드의 마트 표기도 로캘별 (KR: 컬리·쿠팡 등 / US: Walmart·Instacart)
- 금액은 문자열 + 통화 코드 쌍 (float 금지 원칙 정합), 표기 포맷은 프론트 담당
- 모든 텍스트 i18n 키 (`ko.json`/`en.json` 동시), 프롬프트 문구 포함. SEO 메타 로캘별 + hreflang

## 비기능 요구사항
- **성능**: 게스트 홈 LCP < 2.5s, 예산안 적용 갱신은 클라이언트 즉시(< 300ms, 트랜지션 제외)
- **보안**: 아래 "보안 요구사항" 절 — 게스트 입력 서버 전송 금지(가입 시 이전 1회 제외)가 핵심
- **접근성**: 바텀시트 포커스 트랩·ESC 닫기, 스테퍼/선택지 키보드 조작, 금액 통화 포함 낭독, WCAG AA. 프롬프트가 스크린리더 탐색을 강탈하지 않도록 비모달 알림 우선
- **UX 절제**: 프롬프트류(예산 제안·자동주문 제안)는 세션당 각 1회 한도

## UX 흐름 (화면 단위)

```
[/ 게스트 홈] — "체험 모드" 배지, 샘플 식단·예산 무드·냉장고·자동주문 카드
  │  (자유 둘러보기)
  ├─ 체류 10초 + 스크롤 유휴
  │    └─ 바텀시트 "예산안을 작성해 보시겠어요?"
  │         ├─ 아니오 → 닫힘 (세션 내 재노출 없음) + 상단 상시 CTA
  │         └─ 예 ↓
  │    [예산안 작성 오버레이: ① 인원 → ② 월 예산 → ③ 식단 방향]
  │         ↓ 적용 (로딩 트랜지션)
  │    [/ 게스트 홈 — 내 예산안 반영] (여전히 체험 모드, "예시" 라벨 유지)
  │         ↓ 자동주문 카드 활성 + "자동주문을 시작해볼까요?" 알림 1회
  │         └─ 시작하기 ↓
  ├─ (기타 쓰기 행동: 식사 완료 체크 등) → 공통 가입 유도 모달 ↓
[/login 소셜 로그인] (기존 기획)
      ↓ 가입/로그인 완료
  게스트 예산안 있음 → 서버 재검증·계정 이전 → 온보딩 스킵(확인 1장) → [로그인 홈]
  게스트 예산안 없음 → [/onboarding 가구 구성] → [로그인 홈]
```

## 데이터 요구사항 (신규/변경 엔티티, 보관 기간)
- **DB 신규/변경 엔티티 없음, 마이그레이션 없음 (GATE 3 비대상)** — 이전된 예산안의 저장처는 budget/household 도메인 기획·설계에서 확정
- 게스트 예산안 스키마(인원·예산·통화·식단 방향)는 budget/household 실제 스키마와 **필드 정합** 유지 — 이전 시 무손실 매핑
- 정적 샘플 매트릭스(프론트 번들 JSON): 가구 구간(1/2/3~4/5+) × 예산 구간 × 식단 방향(4) × 로캘(ko/en) → {주간 식단, 예산 무드 수치, 냉장고 프리뷰, 주문 추천}. 금액은 문자열+통화 코드
