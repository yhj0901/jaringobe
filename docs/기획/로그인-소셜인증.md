# 로그인 페이지 (소셜 인증) 기획서

## 개요 | 배경 및 목적

자린고비 서비스의 최초 진입점인 **로그인 페이지**와 그 기반이 되는 **소셜 로그인(카카오·구글·애플) 인증 체계**를 만든다.
사업계획서 프로토타입 검증 항목 "소셜 로그인 (카카오 · 구글 · 애플)"에 해당하며, 선순환 루프의 ① 입력(가족·예산 락 설정) 단계로 들어가기 위한 전제 기능이다.

- 이메일/비밀번호 회원가입은 제공하지 않는다 — 소셜 로그인 전용 (가입 마찰 최소화 + 비밀번호 관리 리스크 제거)
- frontend(Next.js)·backend(FastAPI) 초기 스캐폴딩이 선행 의존성이다
- **첫 진입점 변경 (2026-07-09)**: 서비스 첫 화면은 로그인이 아니라 **게스트 홈(`/`)** 이다 (`docs/기획/게스트홈-진입경험.md`). 로그인 페이지(`/login`)는 게스트 홈의 자동주문 전환·가입 게이트 또는 보호 페이지 리다이렉트를 통해 진입한다

## 범위

### In Scope
- 로그인 페이지 UI (브랜드 영역 + 소셜 로그인 버튼, ko/en, 모바일 우선 반응형)
- 카카오·구글 OAuth 로그인 (Authorization Code 방식, 백엔드 토큰 교환) — **P0**
- 애플 로그인 — **P1** (어댑터 인터페이스는 P0 에서 3사 공통으로 확정, 구현만 후행)
- 신규 가입 시 users 자동 생성 (별도 가입 폼 없음)
- JWT 발급 (Access + Refresh) 및 로그아웃
- 로그인 후 분기: 신규 유저(온보딩 미완료) → 가구 구성 온보딩 / 기존 유저 → 홈
- 인증 실패·동의 거부·중복 이메일 안내 UX

### Out of Scope
- 이메일/비밀번호 가입, 아이디/비밀번호 찾기
- 계정 간 연결(하나의 유저에 복수 provider 연결) — 데이터 모델만 대비
- 회원 탈퇴, 프로필 수정 (별도 기획)
- 온보딩(가구 구성) 화면 자체 — 로그인 후 이동만 정의 (별도 기획: household)
- 게스트 홈/예산안 체험 — 별도 기획 (`docs/기획/게스트홈-진입경험.md`)

## 영향 영역
- [x] frontend (Next.js) — 로그인 페이지, 인증 상태 관리, 보호 라우팅
- [x] backend (FastAPI) — OAuth 콜백, provider 어댑터, JWT 발급/갱신/폐기
- [x] DB (PostgreSQL 스키마 변경) — `users`, `auth_identities`, `refresh_tokens` 신규 → **GATE 3 대상**
- [ ] 마트 연동 (store 어댑터) — 해당 없음

## 관련 도메인
- **auth** (중심)
- household (로그인 후 온보딩 분기 지점 — 접점만)

## BM 연결점
- 모든 수익원(중개 수수료·프리미엄 구독·광고·자체 리테일)의 전제인 **회원 식별·계정 체계**를 구축
- 구독(subscription)·주문(order) 도메인이 이 유저 계정에 연결됨

## 글로벌 고려 (KR/US 동작 차이, i18n, 통화)
- **provider 노출 순서 로캘별 상이**: ko → 카카오, 구글, 애플 / en → 구글, 애플 (카카오는 en 로캘에서 숨김 또는 최하단)
- 모든 UI 텍스트 i18n 키 관리 (`ko.json` / `en.json` 동시 작성), 하드코딩 금지
- `users` 에 `locale`, `country`, `currency` 를 처음부터 보유 (기본값: 가입 시 로캘 기반 ko-KR-KRW / en-US-USD)
- 애플 로그인은 iOS 앱 출시 시 스토어 정책상 의무 → **iOS 앱 출시 GATE 전 P1 구현 완료 필수**
- 시각은 UTC 저장, 표시 포맷은 프론트 담당

## 비기능 요구사항
- **성능**: 로그인 페이지 LCP < 2.5s (첫 진입점), OAuth 콜백 처리 p95 < 1s (외부 provider 지연 제외)
- **보안**: 아래 "보안 요구사항" 절 참조
- **접근성**: 소셜 버튼 키보드 조작 가능, 버튼 텍스트 스크린리더 노출 (아이콘 단독 금지), 색 대비 WCAG AA
- **가용성**: 특정 provider 장애 시 다른 provider 로그인은 정상 동작 (어댑터 격리)

## UX 흐름 (화면 단위)

```
[/ 게스트 홈] (별도 기획)
      ↓ 자동주문 시작 / 가입 게이트 행동 / 보호 페이지 접근 리다이렉트
[로그인 페이지 /login]
  ├─ 브랜드: 로고 + "GO BE SMART, SAVE BIG" (i18n)
  ├─ 소셜 버튼 (로캘별 순서)
  └─ 약관/개인정보처리방침 링크 (문서는 별도 준비 — 미결 아님, 법무 산출물)
      ↓ 버튼 클릭
[Provider 동의 화면 (외부)]
      ↓ 승인                    ↓ 거부/실패
[콜백 처리 (로딩 스피너)]      [로그인 페이지 + 안내 배너, 재시도]
      ↓
  신규 (온보딩 미완료) ─┬→ 게스트 예산안 있음: 서버 재검증·계정 이전 → 온보딩 스킵 (게스트 홈 기획 FR-108)
                        └→ 게스트 예산안 없음: [/onboarding 가구 구성]
  기존 ──────────────→ [/home]  (보호 페이지에서 튕겨온 경우 원래 목적지로)
```

## 필요 API 목록 (초안 — 상세는 설계 단계)

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/api/v1/auth/{provider}/authorize` | provider 인가 URL 생성(+`state`) 및 리다이렉트 |
| GET/POST | `/api/v1/auth/{provider}/callback` | code 교환 → 유저 조회/생성 → JWT 발급 |
| POST | `/api/v1/auth/refresh` | Refresh 토큰으로 Access 재발급 (회전) |
| POST | `/api/v1/auth/logout` | Refresh 토큰 폐기 |
| GET | `/api/v1/users/me` | 내 정보 + 온보딩 완료 여부 (분기용) |

- provider ∈ {kakao, google, apple} — 공통 어댑터 인터페이스 뒤로 격리
- 확정 스펙은 설계 단계에서 `docs/설계/api-spec.md` 에 기록 (프론트↔백 계약)

| CWE-601 | Open Redirect | 콜백 후 리다이렉트 목적지는 화이트리스트(자체 경로)만 허용 |

| CWE-613 | 세션 만료 부재 | Access 단기 + Refresh 만료·회전·로그아웃 시 서버측 폐기 |
| CWE-798 / CWE-522 | 자격증명 하드코딩/보호 미흡 | provider 클라이언트 시크릿은 `.env` 전용, 코드/로그 노출 금지 |

- **개인정보 최소 수집**: provider ID·닉네임 외 전부 선택 항목. 생년월일·전화번호 등 수집 안 함
- provider Access Token 은 프로필 조회 후 즉시 폐기 (저장 금지 — 마트 연동 자격증명과 무관)
- 글로벌 대응: 가입 시점 동의 기록(약관·개인정보 버전) 저장 → GDPR/CCPA 기본 대비

## 데이터 요구사항 (신규/변경 엔티티, 보관 기간)

| 엔티티 | 성격 | 주요 필드 (초안) |
|--------|------|------------------|
| `users` | 신규 | id, nickname, email(null 허용), profile_image_url, locale, country, currency, onboarding_completed_at, created_at(UTC) |
| `auth_identities` | 신규 | id, user_id(FK), provider, provider_user_id, email_at_signup — **(provider, provider_user_id) unique**. 1 유저 : N provider 구조로 계정 연결 확장 대비 |
| `refresh_tokens` | 신규 | id, user_id(FK), token_hash, expires_at, revoked_at — 원문 저장 금지(해시) |

- 마이그레이션: `backend/alembic/` 최초 리비전 — **인프라 에이전트 전담, GATE 3 검증 대상**
- 보관 기간: refresh_tokens 는 만료 후 배치 삭제. users 보관/탈퇴 파기 정책은 회원 탈퇴 기획에서 확정 (이번 범위에선 수집 항목 최소화로 리스크 축소)
- 성능: 첫 기능이라 이슈 없음. `auth_identities(provider, provider_user_id)` 유니크 인덱스가 로그인 조회 커버

## 기술 제약사항
- **선행 의존성**: frontend(Next.js App Router + next-intl)·backend(FastAPI + SQLAlchemy async)·docker-compose(postgres) 초기 스캐폴딩이 없음 → 이번 구현 범위에 포함 (백엔드/프론트/인프라 에이전트 분담)
- OAuth 는 Authorization Code 방식으로 3사 통일, provider 어댑터 패턴으로 격리 (애플 P1 추가 시 인터페이스 변경 없어야 함)
- 애플 로그인은 Apple Developer Program(유료) 등록 + 도메인 검증 필요 → P1 배치 근거. 단 **iOS 앱 출시 전 필수 완료**
- 카카오 이메일은 선택 동의 항목 → 이메일 null 허용을 전 도메인이 전제해야 함
- 로컬 개발용 provider 콜백 URL(localhost) 등록 필요 — `.env.example` 에 키 목록 명시
