# 회원 홈 식단 연결 기획서

## 개요 | 배경 및 목적

로그인한 회원의 홈(`/`)을 게스트 샘플 데이터에서 **내 예산안 기반 실제 AI 식단**으로 전환한다. 팀원이 구현한 mealplan 백엔드(생성/조회/재생성, `docs/기획/예산락-AI식단-자동생성.md` — 미머지 문서는 설계 단계에서 회수)의 **프론트 소비자 접점**을 완성해 선순환 루프 ②(AI 식단) 단계를 사용자에게 연다.

- 전제: v0.1.0 (게스트 홈 + 소셜 로그인 + 예산안 이전) 프로덕션 라이브, mealplan API 3종 배포됨
- 홈 셸은 `HomeViewModel` 주입형으로 설계되어 있어 **member 모드 데이터 어댑터 추가**가 핵심

## 범위

### In Scope
- 회원 홈(mode='member'): 내 최신 식단 조회 → 주간 식단·예산 요약 표시 (오늘 기본, 주간 스트립 이동)
- 식단 없음(빈 상태): 예산 락 히어로 + "내 식단 만들기" 대형 CTA → 생성 시트(기간·알레르기·선호 선택 입력) → `POST /mealplans` → 로딩 연출 → 표시
- 예산 요약 연동: `budgetSummary`(편성 비용/남은 예산/withinBudget) → 예산 무드 카드. **예산 초과 시 초과 배너 + 재생성 유도**
- 식단 재생성(전체) — P1
- 게스트 예산안 없이 가입한 회원: 홈에서 예산안 작성 유도 (기존 `BudgetDraftFlow` 재사용, `POST /budget/plans source='onboarding'` 서버 저장)
- 냉장고 위젯·자동주문 카드는 회원에게 **"준비 중" 잠금 상태** 표시 (게스트 샘플 노출 금지)
- **필요 백엔드 추가 1건**: 내 최신 식단 조회 API (아래 "필요 API")

### Out of Scope
- 식단 생성 로직·예산 검산·기준가 자체 (팀원 기획 FR-001~008 — 변경 요청 없음)
- 식사 완료 체크·냉장고 자동 차감 (fridge 기획), 자동 주문/마트 연동 (**feature/store 작업과 겹침 방지** — order/store 일절 제외)
- 끼니 단위 재생성(P2), 조리법 상세 화면(레시피 단계 데이터 부재), 가구 구성 온보딩 화면(household 기획)
- 푸시/알림

## 영향 영역
- [x] frontend (Next.js) — member 데이터 어댑터, 빈 상태/생성 시트/로딩/초과 배너, 카드 잠금 상태
- [x] backend (FastAPI) — **최신 식단 조회 API 1건 추가** (mealplan 도메인, 팀원 협의)
- [ ] DB — 변경 없음 (기존 테이블 조회) → **GATE 3 비대상**
- [ ] 마트 연동 — 없음

## 관련 도메인
- **mealplan** (중심), budget (예산 무드·예산안 작성 유도), auth (로그인 후 분기 접점)

## BM 연결점
- AI 식단이 실제로 보이는 순간이 **프리미엄 식단 구독**의 가치 증명 지점. 예산 요약(절약 지표)은 유지·리텐션 장치
- 식단 → (후속) 자동 주문으로 이어지는 중개 수수료 퍼널의 중간 단계 완성

## 글로벌 고려 (KR/US 동작 차이, i18n, 통화)
- API 가 `region`/`currency` 를 내려줌 — 금액 표시는 기존 `MoneyText`(Intl) 재사용, KRW/USD 자동
- 생성 시트·로딩 문구·초과 배너 등 신규 텍스트 전부 i18n 키 (ko/en 동시)
- 알레르기/선호 입력은 자유 텍스트(로캘 무관, LLM 전달) — placeholder 만 로캘별

## 비기능 요구사항
- **성능**: 홈 진입 시 최신 식단 조회 p95 < 500ms(서버), 생성은 비동기 UX 로 체감 완화 (LLM 수 초~수십 초 허용)
- **보안**: 아래 보안 요구사항 절
- **접근성**: 로딩 상태 aria-busy/라이브 리전, 생성 시트 키보드 조작, 초과 배너 색+아이콘 병행(색 단독 금지)
- **비용 방어**: 생성·재생성은 기존 유저 rate limit(5/분) 전제 — 프론트는 연타 방지(버튼 비활성)

## UX 흐름 (화면 단위)

```
[로그인 완료 → / 홈 (member)]
  ├─ 예산안 없음 → 안내 카드 → [BudgetDraftFlow] → POST /budget/plans(onboarding) ↓
  ├─ 최신 식단 없음(404) → [빈 상태: 예산 락 히어로 + "내 식단 만들기"]
  │     └─ [생성 시트: 기간/알레르기/선호] → POST /mealplans
  │           → [로딩: 단계 문구+스켈레톤] → 성공 ↓ / 실패·429 → 배너+재시도
  └─ 최신 식단 있음 → [주간 식단 홈]
        ├─ 예산 무드(편성/남은/절약) — withinBudget=false 시 초과 배너 → 재생성 유도
        ├─ 주간 스트립(일자 이동) + 끼니 행(재료·추정 비용)
        ├─ 재생성 버튼(P1, 확인 다이얼로그)
        └─ 냉장고/자동주문 카드: "준비 중" 잠금
```

## 데이터 요구사항 (신규/변경 엔티티, 보관 기간)
- **DB 신규/변경 없음 → GATE 3 비대상.** latest 조회는 `meal_plans(user_id, created_at)` 기존 구조로 충분 (필요 시 인덱스는 설계에서 판단)
- HomeViewModel 확장: member 모드 필드(주간 스트립 일자, 초과 여부 등)는 **옵셔널 추가만** (게스트 계약 파괴 금지)

## 기술 제약사항
- 홈 셸·위젯 컴포넌트는 리스타일 완료 상태 — member 모드도 동일 컴포넌트 재사용 (디자인 프로토타입의 "식사 완료" CTA 는 fridge 기획에서)
- 최신 식단 조회 전까지 프론트는 스켈레톤 — plan id 로컬 저장 금지(기기 간 동기화)
- 생성 요청 파라미터는 팀원 스키마(`days/mealsPerDay/allergies/preferences`) 준수 — 임의 확장 금지
- feature/store 브랜치와 파일 겹침 없도록: `features/order`·`features/store` 미생성, AutoOrderCard 는 잠금 상태 표시만
