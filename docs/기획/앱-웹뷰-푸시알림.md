# 앱 웹뷰 래핑 + 푸시 알림 기획서

## 개요 | 배경 및 목적

기존 Next.js 웹 서비스를 모바일 앱(웹뷰 쉘)으로 감싸 iOS/Android 스토어에 출시하고, 푸시 알림 채널을 확보한다.

**배경 (해결하려는 문제)**
1. **식단 생성 대기 문제**: AI 식단 생성은 시간이 오래 걸린다. 현재 `POST /api/v1/mealplans` 는 동기 방식이라 사용자가 화면을 붙잡고 기다려야 한다. → 생성을 비동기화하고 완료 시 푸시로 알린다.
2. **식사 리마인더 부재**: "지금 만들어 먹을 시간" 알림이 없어 식단이 있어도 식사 완료 체크(→ 가상 냉장고 자동 차감)로 이어지지 않는다. → 끼니별 리마인더 푸시로 선순환 루프 ③(가상 냉장고 차감)④(동적 감산)를 강화한다.

**제품 방향 정합성**: 푸시는 그 자체가 3대 축은 아니지만, 루프의 회전율(식사 완료 체크율·재방문율)을 높이는 **루프 가속 장치**다. 웹만으로는 iOS 푸시 도달이 사실상 불가(홈 화면 추가 PWA 한정)하므로 앱 쉘이 전제 조건이다.

## 범위: In Scope / Out of Scope

### In Scope
- 모바일 앱 쉘: **Expo(React Native) + react-native-webview** — 배포된 웹 URL 을 로드하는 얇은 껍데기 (신규 `mobile/` 디렉토리)
- 푸시 알림 2종:
  - **식단 생성 완료 알림** (이벤트성) — 생성 비동기 전환 포함
  - **식사 시간 리마인더** (스케줄성, 아침/점심/저녁 개별 설정)
- 웹 ↔ 앱 브리지 (postMessage JSON 프로토콜, 버저닝)
- 디바이스 토큰 등록/갱신/해제 API 및 백엔드 발송 서비스 (**신규 `notification` 도메인**)
- 알림 설정 화면 (웹 UI — 유형별 on/off, 리마인더 시각 변경)
- 푸시 탭 → 딥링크 → 웹뷰 내 해당 경로 이동
- 소셜 로그인 웹뷰 대응 (구글: 시스템 브라우저/커스텀 탭 + 앱 복귀)
- 식단 생성 비동기 전환에 따른 **웹 폴링 병행** (앱 미설치 웹 사용자 대응)
- 푸시 메시지 i18n (ko/en)

### Out of Scope (차기)
- 광고성/마케팅 푸시 (도입 시 별도 수신 동의 체계 + 야간 전송 제한 필요 — 아래 보안 요구사항 참조)
- 유통기한 임박 재료 푸시 (현재 인앱 배너로 대체 중 — 차기 확장 1순위)
- 주문/배송 상태 푸시 (order 도메인 구현 진도에 종속)
- 네이티브 화면 개발 (웹뷰 유지가 원칙 — 단 Apple 심사 Plan B 로 네이티브 탭바만 예외, 기술 제약사항 참조)
- 브라우저용 웹 푸시 (Web Push API)
- 태블릿 최적화

## 영향 영역
- [x] frontend (Next.js) — 브리지 수신/토큰 전달, 알림 설정 UI, 생성 진행 상태 폴링 UI
- [x] backend (FastAPI) — 신규 notification 도메인, mealplan 생성 비동기 전환, 리마인더 스케줄러
- [x] DB (PostgreSQL 스키마 변경) — 신규 3테이블 + mealplans.status 컬럼 (GATE 3 대상)
- [ ] 마트 연동 (store 어댑터) — 영향 없음
- [x] **mobile/ (신규 영역)** — Expo RN 웹뷰 쉘 (담당: UI 프론트엔드 에이전트 관할 확장, 기술 제약사항 참조)

## 관련 도메인
- **notification (신규)** — 디바이스 토큰, 알림 설정, 발송 서비스, 스케줄러
- **mealplan** — 생성 비동기 전환(status), 완료 이벤트 발행, 리마인더의 발송 조건(당일 식단·완료 여부)
- **auth** — 웹뷰 내 소셜 로그인 대응(구글 커스텀 탭), 로그아웃 시 토큰 해제, 애플 로그인 P1 의존성

## BM 연결점
- **프리미엄 구독**: 푸시 → 재방문·식사 완료 체크율 상승 → 루프 체감 가치 상승 → 구독 전환 기반. 차기에 프리미엄 전용 알림(다이어트 코칭 등) 확장 여지
- **중개 수수료**: 리마인더 → 식사 완료 → 냉장고 차감 → 동적 감산 → 다음 자동 주문 정확도 상승 (간접 기여)

## 글로벌 고려 (KR/US 동작 차이, i18n, 통화)
- **발송 인프라 공통**: Expo Push Service 는 FCM(Android)·APNs(iOS)를 단일 API 로 추상화 — KR/US 국가별 어댑터 불필요
- **타임존**: 리마인더는 사용자 타임존 기준 로컬 시각으로 발송. 설정은 "로컬 시각 + IANA 타임존" 쌍으로 저장(DST 대응), 발송 큐는 UTC 환산 컬럼으로 관리
- **i18n**: 푸시 제목/본문은 사용자 로캘(ko/en)로 발송 — 메시지 템플릿을 i18n 키로 관리, ko/en 동시 작성
- **로그인**: en 로캘(미국) 사용자는 구글/애플 중심 → 웹뷰 구글 OAuth 대응이 KR 보다 더 치명적. 애플 로그인(P1)은 iOS 출시 전 완료 필수 (기존 기획 `로그인-소셜인증.md` 의무 조항)
- **통화**: 이번 기능은 금액 표시 없음 — 푸시 본문에 예산액 노출 금지(보안 요구사항)와 일관

## 비기능 요구사항
- **성능**: 생성 완료 푸시는 작업 완료 후 30초 이내 발송. 리마인더는 설정 시각 ±2분 이내. 웹 폴링 주기는 3~5초(백오프 포함)
- **신뢰성**: 푸시는 보조 채널 — 미도달해도 웹/앱 화면에서 동일 정보 확인 가능해야 함 (푸시에만 의존하는 흐름 금지)
- **스케줄러 확장성**: 타임존별 매분 전체 스캔 금지 — "다음 발송 시각(UTC)" 컬럼 + 인덱스 스캔 방식
- **접근성**: 알림 설정 UI 는 기존 웹 접근성 기준 준수. 푸시 본문은 스크린리더로 읽혀도 자연스러운 완결 문장
- **앱 크기/유지비**: 쉘은 얇게 유지 — 네이티브 코드 추가는 푸시·스플래시·브리지·(Plan B 탭바)로 한정

## UX 흐름 (화면 단위)
1. **앱 최초 실행**: 네이티브 스플래시 → 웹뷰 로드(게스트 홈 `/`) → 기존 웹 흐름 그대로 (로그인/온보딩)
2. **권한 요청**: 첫 식단 생성 요청 직후 soft ask 모달("완성되면 알려드릴까요?") → 수락 시 OS 권한 다이얼로그 → 토큰 발급 → 브리지로 웹 전달 → 서버 등록
3. **식단 생성**: 요청 → 즉시 "생성 중" 화면(진행 상태 폴링, 웹/앱 공통) → 사용자는 이탈 가능 → 완료 푸시 "🍽 이번 주 식단이 완성됐어요" → 탭 → 식단 상세
4. **식사 리마인더**: 설정 시각 도래 + 당일 끼니 식단 존재 + 미완료 → "오늘 점심: 김치찌개 — 지금 만들어 볼까요?" → 탭 → 해당 식단/레시피 → 식사 완료 체크 → 냉장고 자동 차감 (루프 ③)
5. **알림 설정**: 설정 페이지 > 알림 — 유형별 토글, 끼니별 시각, OS 권한 꺼짐 배너
6. **로그인(앱 내)**: 로그인 버튼 → 구글은 커스텀 탭/시스템 브라우저 → 완료 후 앱 딥링크 복귀 → 웹뷰 세션 반영

## 데이터 요구사항 (신규/변경 엔티티, 보관 기간)

| 엔티티 | 구분 | 주요 필드 | 보관 |
|--------|------|-----------|------|
| `device_tokens` | 신규 | user_id FK, platform(ios/android), token(unique), locale, timezone, app_version, last_seen_at | 계정 삭제 시 파기, 무효 토큰 즉시 삭제 |
| `notification_settings` | 신규 | user_id FK, type(meal_reminder_breakfast/lunch/dinner, mealplan_done, …), enabled, local_time, timezone(IANA), next_send_at(UTC, 인덱스) | 계정과 수명 동일 |
| `notification_logs` | 신규 | user_id, device_token_id, type, title/body 스냅샷, status(sent/failed), sent_at | **90일** 후 배치 삭제 |
| `mealplans.status` | 변경(컬럼 추가) | pending / processing / completed / failed | 기존 테이블 수명 동일 |

**마이그레이션 리스크 (데이터 검토관 기록)**: 4건 모두 additive(신규 테이블 3 + 컬럼 1) — 파괴적 변경 없음, 리스크 낮음. `mealplans.status` 는 기존 행 기본값 `completed` 백필 필요. 리마인더 발송은 `next_send_at` UTC 인덱스 스캔으로 처리(타임존별 전체 스캔 금지). GATE 3 승인 대상, DDL 은 인프라 에이전트 전담.

## 기술 제약사항
1. **스택 확정**: `mobile/` = Expo(React Native) + react-native-webview + expo-notifications + Expo Push Service. EAS Build/Submit 로 스토어 제출. Capacitor 는 Next.js SSR 원격 URL 모드의 제약으로 기각
2. **비동기 실행 방식**: MVP 는 FastAPI BackgroundTasks(단일 프로세스)로 시작, 워커 분리/큐 도입은 부하 확인 후 — 최종 방식은 설계 단계(GATE 2)에서 확정
3. **Apple 심사 4.2 (최소 기능성) 리스크**: 1차 방어 = 푸시 + 네이티브 스플래시 + 권한 연동 알림 설정. **리젝 시 Plan B = 네이티브 하단 탭바(Expo Router) 추가** — 별도 스프린트로 기획서 범위 밖 예산 확보
4. **애플 로그인 의존성**: iOS 스토어 정책상 소셜 로그인 제공 앱은 애플 로그인 의무 — 기존 기획(`로그인-소셜인증.md` P1)을 **iOS 출시 전 완료 필수** 로 승격. Android 는 무관하므로 심사 일정상 Android 선출시 가능
5. **구글 OAuth**: 웹뷰 내 진행 불가(정책 차단) — 커스텀 탭/시스템 브라우저 + 앱 스킴 복귀. 카카오 웹뷰 허용 여부는 설계 단계에서 최신 정책 재확인
6. **에이전트 관할**: `mobile/` 은 UI 프론트엔드 에이전트 관할로 확장 — 설계 승인 시 CLAUDE.md 수정 영역 표에 반영 (문서 에이전트)
7. **계정/인프라 선행 조건**: Apple Developer Program(연 $99)·Google Play Console(1회 $25) 계정, Expo 계정(EAS). 코드 착수 전 발급 필요
8. **브리지 하위호환**: 앱 배포는 심사로 느리고 웹은 즉시 배포 — 브리지 프로토콜은 `v` 필드로 버저닝하고 웹이 구버전 앱을 항상 지원
