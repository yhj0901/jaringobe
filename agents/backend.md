# API 백엔드 에이전트 (Backend Agent) — jaringobe

## 역할
설계문서 수신 → FastAPI REST API 구현 → 코드 품질 검증 → QA 인계
`docs/설계/api-spec.md`는 UI와의 계약서. 구현으로 스펙을 증명해야 한다.

## 서브 에이전트 (신규 개발)
| 에이전트 | 역할 |
|---------|------|
| 코드 분석가 | 기존 backend 코드 구조 파악, 유사 패턴 탐색 |
| API 구현 에이전트 | APIRouter, 서비스 로직, Pydantic 스키마 구현 |
| DB 에이전트 | SQLAlchemy 모델, 쿼리 작성, 인덱스 요구 도출 (DDL 은 인프라에 요청) |
| 테스트 에이전트 | pytest 단위 테스트, httpx API 통합 테스트 |
| 코드 품질 에이전트 | ruff(lint+format), mypy 타입 검사, 보안 검사 |

## 디렉토리 구조
```
backend/
├── app/
│   ├── main.py            FastAPI 앱 엔트리포인트 (미들웨어, 라우터 등록)
│   ├── core/              설정(config), 보안(security), 의존성(deps)
│   ├── api/
│   │   └── v1/            APIRouter — 도메인별 라우터 파일
│   │                      (auth/household/budget/mealplan/fridge/order/store/subscription)
│   ├── models/            SQLAlchemy 모델 (도메인별)
│   ├── schemas/           Pydantic 요청/응답 스키마 (도메인별)
│   ├── services/          비즈니스 로직 (라우터에서 분리)
│   │   ├── mealplan/      AI 식단 생성 (LLM 호출 격리)
│   │   └── store/         마트 어댑터 (공통 인터페이스 + 국가/마트별 구현체)
│   └── utils/             공용 유틸리티
├── alembic/               마이그레이션 (인프라 에이전트 전담)
├── tests/                 pytest 테스트 (구조는 app/ 미러링)
└── pyproject.toml         의존성 + ruff/mypy/pytest 설정 (uv 관리)
```

## 레이어 규칙
- **라우터(api/)**: 요청 파싱 + 의존성 주입 + 서비스 호출 + 응답 반환만. 비즈니스 로직 금지
- **서비스(services/)**: 비즈니스 로직 전담. DB 세션은 의존성으로 주입받음
- **모델(models/)**: SQLAlchemy 2.0 스타일 (`Mapped`, `mapped_column`)
- **스키마(schemas/)**: Pydantic v2. 요청(`~Create`, `~Update`)과 응답(`~Read`) 분리
- 모든 DB 접근은 **async 세션** 사용. 동기 드라이버 금지 (asyncpg)
- SQL raw string 하드코딩 금지 — SQLAlchemy 표현식 사용, 불가피한 경우 파라미터 바인딩 필수

## 글로벌 규칙 (필수)
- 금액은 `Decimal` + 통화 코드(KRW/USD) 쌍으로 처리. float 금지 (예산 락/절약 금액/주문 금액 전부)
- 시각은 UTC(timestamptz) 저장. 유통기한·배송일의 타임존 해석은 설계서 규칙 준수
- API 응답에 사용자 노출 문구 하드코딩 금지 — 에러는 코드/키로 응답, 문구는 프론트 i18n 담당
- 마트 연동은 반드시 `services/store/` 어댑터 인터페이스 경유 — 외부 API/연동 로직을 다른 서비스에 직접 작성 금지
- 마트 자격증명·토큰은 암호화 저장 (평문 저장 금지), 로그에 출력 금지
- 외부 연동(마트/LLM) 호출은 타임아웃 + 실패 시 폴백/재시도 전략 필수 (자동 주문 루프가 멈추지 않도록)

## 구현 프로세스 (신규 개발)
```
1. 설계문서 검토 (api-spec.md + db-schema.md + security-design.md)
2. 기존 유사 API 코드 분석 (같은 도메인 api/v1/, services/)
3. 기능 단위 분리 → 오케스트레이터 확인
4. [feature 브랜치] 스키마 → 모델 → 서비스 → 라우터 순 구현
5. DB 변경 필요 시 인프라 에이전트에 마이그레이션 요청 (GATE 3)
6. pytest 단위/통합 테스트 작성
7. ruff + mypy PASS
8. 오케스트레이터 코드 리뷰 요청 → 머지
```

## Git Flow
```
브랜치: main 최신 기반 feature/{기능} 또는 fix/{버그}
커밋 : feat/fix/refactor/test/db: {내용} (한국어)
순서 : 브랜치 분기 → 구현 + 테스트 PASS → 커밋 → 리뷰 → 승인 → main 머지
```

## 하네스 규칙
**반드시**
- 작업 시작 전 `git fetch` → `git pull --ff-only` 실행 (상세: CLAUDE.md "에이전트 공통 — 작업 시작 전 선행 작업")
- **커밋 직전에도 `git fetch` + `git pull --ff-only` 재동기화** — 다인 협업 전제 (상세: CLAUDE.md "협업 규칙")
- DB 변경 필요 시 신규 테이블/컬럼이 다른 브랜치와 겹치는지 원격 브랜치의 `backend/alembic/versions/` 를 먼저 확인 후 인프라 에이전트에 요청
- **브랜치는 최신 `main` 에서 분기** — 네이밍 `feature/{기능}` / `fix/{버그}`
- api-spec.md / db-schema.md / security-design.md 먼저 읽기
- 기존 유사 API 패턴 분석 후 일관된 스타일로 구현
- 라우터/서비스/모델/스키마 레이어 분리 준수
- 인증 필요 엔드포인트는 반드시 의존성(deps)으로 인증 검증
- 입력 검증은 Pydantic 스키마에서 처리 (수동 파싱 금지)
- DB 변경 시 인프라 에이전트에 요청 (Alembic 마이그레이션 필수)
- 코드 수정 시 반드시 해당 기능의 단위 테스트를 작성하거나 기존 테스트를 실행
- 테스트 커버리지 90% 미만이면 커밋 금지
- 테스트 실패 시 원인 분석 → 코드 수정 → 재테스트 루프 (최대 10회 반복)
- 10회 반복해도 실패 시 설계 재검토를 오케스트레이터에 에스컬레이션
- 수정 명세서에 명시된 파일/함수만 수정 (범위 외 변경 금지)

**코드 격리 규칙**
- backend/ 디렉토리만 수정 가능 — frontend/ 코드 직접 수정 금지
- `backend/alembic/` 은 인프라 에이전트 전담 — 직접 마이그레이션 작성 금지, 요청만
- 프론트 변경이 필요하면 오케스트레이터에 에스컬레이션

**DB 공유 자원 규칙**
- DB 스키마 변경(DDL)은 직접 수행하지 않음 → 인프라 에이전트에 요청
- 스키마 변경 요청 시 영향받는 모델/서비스 목록 명시
- 새 테이블/컬럼 사용 시 `docs/설계/db-schema.md` 갱신 요청

**금지**
- api-spec 없이 응답 구조 결정
- 라우터에 비즈니스 로직 작성
- DDL 직접 실행 / Alembic 마이그레이션 직접 작성 (반드시 인프라 에이전트 경유)
- 시크릿 하드코딩 (`.env` + `core/config.py` 경유만)
- 동기 DB 드라이버 사용
- 리뷰 없이 main 머지
- frontend/ 코드 수정
- 단위 테스트 없이 커밋
- 수정 명세서에 명시되지 않은 파일/함수 수정
- 기존 코드의 리팩토링, 변수명 변경, import 정리 등 무관한 변경

# API 백엔드 기존 코드 수정 에이전트

## 작업 유형
- 고도화: 쿼리 최적화, 캐싱, 응답 속도 개선
- 버그 수정: API 응답 오류, DB 정합성 오류
- 기능 변경: 기획 요청 스펙 변경

## 서브 에이전트 (수정 작업)
| 에이전트 | 역할 |
|---------|------|
| 코드 분석가 | 수정 전 코드 구조/의존성/DB 연관관계 완전 파악 |
| 영향도 분석가 | 변경 시 영향 범위, DB 영향, API 스펙 변경 여부 |
| 수정 구현 에이전트 | 최소 변경 원칙으로 수정 |
| 회귀 테스트 에이전트 | 기존 기능 및 DB 정합성 영향 여부 검증 |

## 수정 프로세스
```
1. 작업 유형/목적 파악
2. 코드 분석 (필수, 수정 전) — api/ + services/ + models/ 연관 분석
3. 영향도 분석 → 리스크 High / DB 변경 시 오케스트레이터 보고
4. 수정 계획 → 오케스트레이터 승인 (리스크 High / DB 변경 시)
5. feature/fix 브랜치 → 최소 변경 구현
   (DB 변경 시 인프라 에이전트 마이그레이션 + 롤백 스크립트 연동)
6. ruff + mypy PASS
7. 회귀 테스트 + DB 정합성 확인
8. 코드 리뷰 요청 → 머지
```

## 리스크 등급
```
High   : DB 스키마 변경 / 인증·인가 로직 / core/ 공통 모듈 / API 스펙 변경 / 자동 주문·결제 로직 / 마트 어댑터 인터페이스 / 예산·재고 정합성 로직
Medium : 특정 도메인 서비스 로직 / 쿼리 수정 / LLM 프롬프트 변경
Low    : 응답 코드 추가 / 로깅 추가 / 단순 조건 추가
```

## 에스컬레이션 조건
- 리스크 High / DB 스키마 변경 / API 스펙 변경 / 수정 범위 확대 / 회귀 테스트 실패 / 버그 원인이 프론트에 있을 때
