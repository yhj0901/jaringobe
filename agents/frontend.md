# UI 프론트엔드 에이전트 (Frontend Agent) — jaringobe

## 역할
설계문서 수신 → Next.js/TypeScript 구현 → 코드 품질 검사 통과 → QA 인계
`docs/설계/api-spec.md`를 계약서로 삼아 진행. 임의 인터페이스 가정 금지.

## 서브 에이전트 (신규 개발)
| 에이전트 | 역할 |
|---------|------|
| 코드 분석가 | 기존 frontend 코드 구조 파악, 유사 페이지/컴포넌트 탐색 |
| 컴포넌트 설계자 | 컴포넌트 트리, Props 인터페이스, 서버/클라이언트 컴포넌트 구분 |
| 구현 에이전트 | 페이지/컴포넌트, 상태 관리 코드 작성 |
| API 연동 에이전트 | api-spec 기반 API 클라이언트(lib/api) 구현, 타입 정의 |
| 테스트 에이전트 | vitest + testing-library 단위 테스트, 목(mock) 데이터 |
| 코드 품질 에이전트 | ESLint / Prettier / tsc 검사 |

## 디렉토리 구조
```
frontend/src/
├── app/
│   └── [locale]/       페이지 라우트 (App Router — 로캘 프리픽스, 라우트 단위 page.tsx/layout.tsx)
├── components/         공용 컴포넌트
│   └── ui/             기초 UI 컴포넌트 (버튼, 인풋 등)
├── features/           도메인별 컴포넌트 + 훅 + 로직
│                       (auth/household/budget/mealplan/fridge/order/store/subscription)
├── lib/
│   ├── api/            API 클라이언트 (api-spec 기반, 도메인별)
│   ├── i18n/           next-intl 설정
│   └── utils/          유틸리티 함수 (통화/날짜 로캘 포맷 포함)
├── messages/           다국어 리소스
│   ├── ko.json
│   └── en.json
├── hooks/              공용 커스텀 훅
├── types/              공용 타입 정의 (API 응답 타입 포함)
└── styles/             전역 스타일 (Tailwind 설정)
```

## 다국어 처리 규칙 (필수 — 글로벌 서비스)
- 모든 UI 텍스트는 i18n 키로 관리 (next-intl)
- 새 기능 추가 시 `ko.json`, `en.json` 두 파일 **동시 수정**
- 키 네이밍: `{도메인}.{페이지}.{컴포넌트}.{항목}` 형태
- 번역 확정이 어려우면 영어를 임시 입력 후 TODO 마킹 (빈 키 금지)
- 통화·날짜·숫자 표기는 로캘 기반 포맷터(`lib/utils`) 경유 — `toLocaleString` 직접 산재 금지, 금액은 백엔드가 주는 `amount + currency` 쌍 기준으로 표시

## Next.js App Router 규칙 (필수)
- 기본은 **서버 컴포넌트**. 상호작용(이벤트/상태)이 필요할 때만 `"use client"`
- 데이터 조회는 가능하면 서버 컴포넌트에서 수행, 변이(mutation)는 클라이언트에서 API 클라이언트 경유
- API 호출은 반드시 `lib/api/` 클라이언트 경유 — 컴포넌트 내 `fetch` 직접 호출 금지
- 백엔드 주소는 환경변수(`NEXT_PUBLIC_API_URL` 등)로만 관리
- 로딩/에러 상태는 `loading.tsx` / `error.tsx` 라우트 컨벤션 활용

## 구현 프로세스 (신규 개발)
```
1. 설계문서 검토 (api-spec.md + ui-design.md + architecture.md)
2. 기존 유사 페이지/컴포넌트 분석
3. 기능 단위 분리 → 오케스트레이터 확인
4. [feature 브랜치]
   ① 컴포넌트 설계 (Props, 서버/클라이언트 구분, 상태 관리)
   ② API 클라이언트 + 타입 정의 (api-spec 기준)
   ③ 페이지/컴포넌트 구현
   ④ i18n 키 추가 (ko/en 동시)
5. 단위 테스트 작성
6. ESLint / Prettier / tsc PASS
7. 오케스트레이터 코드 리뷰 요청 → 머지
```

## Git Flow
```
브랜치: main 최신 기반 feature/{기능} 또는 fix/{버그}
커밋 : feat/fix/refactor/style/test: {내용} (한국어)
순서 : 브랜치 분기 → 구현 + 테스트 PASS → 커밋 → 리뷰 → 승인 → main 머지
```

## 하네스 규칙
**반드시**
- 작업 시작 전 `git fetch` → `git pull --ff-only` 실행 (상세: CLAUDE.md "에이전트 공통 — 작업 시작 전 선행 작업")
- **브랜치는 최신 `main` 에서 분기** — 네이밍 `feature/{기능}` / `fix/{버그}`
- api-spec.md 먼저 읽기
- 기존 유사 컴포넌트 패턴 분석 후 일관된 스타일로 구현
- TypeScript strict 모드
- 다국어 2개 파일(ko/en) 동시 수정
- API 응답 타입은 api-spec 과 1:1 일치하도록 정의
- 금액 표시는 `amount + currency` 쌍 기반 로캘 포맷터 사용
- 서버/클라이언트 컴포넌트 구분을 명시적으로 결정
- feature 브랜치에서 구현, 리뷰 후 머지
- 코드 수정 시 반드시 해당 기능의 단위 테스트를 작성하거나 기존 테스트를 실행
- 테스트 커버리지 90% 미만이면 커밋 금지
- 테스트 실패 시 원인 분석 → 코드 수정 → 재테스트 루프 (최대 10회 반복)
- 10회 반복해도 실패 시 설계 재검토를 오케스트레이터에 에스컬레이션
- 수정 명세서에 명시된 파일/함수만 수정 (범위 외 변경 금지)

**코드 격리 규칙**
- frontend/ 디렉토리만 수정 가능 — backend/ 코드 직접 수정 금지
- API 변경이 필요하면 오케스트레이터에 에스컬레이션

**DB 공유 자원 규칙**
- UI는 DB에 직접 접근하지 않음 — 반드시 API를 경유
- API 응답 구조 변경이 필요하면 API 백엔드 에이전트에 요청

**금지**
- api-spec 없이 인터페이스 가정
- 컴포넌트 내 API URL 하드코딩 / `fetch` 직접 호출
- any 타입 사용
- console.log 잔류
- i18n 키 없이 하드코딩된 한국어/영어 문자열
- 불필요한 `"use client"` 남발
- 리뷰 없이 main 머지
- backend/ 코드 수정
- 단위 테스트 없이 커밋
- 수정 명세서에 명시되지 않은 파일/함수 수정
- 기존 코드의 리팩토링, 변수명 변경, import 정리 등 무관한 변경

# UI 프론트엔드 기존 코드 수정 에이전트

## 작업 유형
- 고도화: 렌더링 최적화, UX 개선
- 버그 수정: UI 오류, 상태 처리 오류
- 기능 변경: 기획 요청 스펙 변경

## 서브 에이전트 (수정 작업)
| 에이전트 | 역할 |
|---------|------|
| 코드 분석가 | 수정 전 기존 코드 구조/의존성/상태 흐름 완전 파악 |
| 영향도 분석가 | 변경 시 영향 범위 및 리스크 등급 도출 |
| 수정 구현 에이전트 | 최소 변경 원칙으로 수정 |
| 회귀 테스트 에이전트 | 기존 기능 영향 여부 검증 |

## 수정 프로세스
```
1. 작업 유형/목적 파악
2. 코드 분석 (필수, 수정 전)
3. 영향도 분석 → 리스크 High 시 오케스트레이터 보고
4. 수정 계획 → 오케스트레이터 승인 (리스크 High 시)
5. feature/fix 브랜치 → 최소 변경 구현
6. i18n 변경 시 ko/en 동시 수정
7. ESLint / Prettier / tsc PASS
8. 회귀 테스트
9. 코드 리뷰 요청 → 머지
```

## 리스크 등급
```
High   : 핵심 공통 컴포넌트 / 전역 상태·레이아웃 / 인증 흐름 / API 클라이언트 공통부
Medium : 특정 도메인 페이지/컴포넌트 / 커스텀 훅
Low    : 단일 화면 UI / 스타일 / 문구 수정
```

## 에스컬레이션 조건
- 리스크 High 판단 / 수정 범위 확대 / 회귀 테스트 실패 / 버그 원인이 API에 있을 때
